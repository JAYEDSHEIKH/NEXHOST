Server API design (simple, JSON over HTTP)
- Base: GET/POST JSON on configurable port (default 8080). Documented in README.

Endpoints (examples)
- GET /api/v1/instances
  - Response: JSON array of instances (id, name, type, version, status, path, xms, xmx, createdAt)
- POST /api/v1/instances
  - Request JSON:
    { "name":"test1", "type":"paper", "version":"latest" | "1.21.4", "xms":"512M", "xmx":"512M", "acceptEula":true|false }
  - Behavior:
    - Create directory under serversRoot/<uuid>/
    - Save instance metadata into instance.properties (or JSON)
    - If acceptEula==true -> write eula.txt with "eula=true"
    - Trigger download of server.jar (latest if version==latest)
    - Return instance id + 202 Accepted (download may be async). Provide progress via /api/v1/instances/{id}/status or via SSE.
- GET /api/v1/instances/{id}
  - Return metadata + status
- POST /api/v1/instances/{id}/start
  - Starts server process (if jar present and eula accepted). Respond 200 on success.
- POST /api/v1/instances/{id}/stop
  - Sends save-all and stop commands; force-kill after timeout.
- GET /api/v1/instances/{id}/console/stream
  - Server-Sent Events (SSE) or chunked responses providing real-time log lines.
- POST /api/v1/instances/{id}/console/command
  - Body: { "command":"list" } — writes to server stdin.
- DELETE /api/v1/instances/{id}
  - Stops and deletes all files for instance (with confirmation).

Server-prototype implementation notes
- Implement ServerApi using com.sun.net.httpserver.HttpServer (lightweight) or Jetty if you prefer. Keep dependency minimal for Replit.
- Store instance metadata as instance.properties JSON in each instance folder (no DB). Provide atomic write/rename to avoid corruption.
- Downloads:
  - For Paper: query PaperMC API v2 to find latest build URL for a version. If user passes "latest" -> pick latest stable.
  - For Vanilla: use Mojang version manifest to find server jar URL.
  - Always compute SHA-256 after download and store as checksum field. If provider returns hash, validate.
  - Timeouts, retries, and partial-download resume are optional but recommended (3 attempts).
- Processes:
  - Build command with normalized memory: append "M" to plaintext numeric values.
  - Use ProcessBuilder to start java path found via System.getProperty("java.home") or use "java" fallback.
  - Ensure process working dir is instance folder.
  - Stream console stdout/stderr via background threads; append to rolling log file and also push lines to SSE clients.
  - When stopping: send "save-all\n", wait 2s, then "stop\n", wait up to 30s, then destroyForcibly.
- Concurrency:
  - Limit concurrent server starts to a safe number (configurable) to avoid resource exhaustion on small hosts.
- Logging:
  - Write logs into instance/logs/server-<timestamp>.log. Expose the last N lines via API and SSE for streaming.

2) Memory normalizer (ServerProcessManager change):

```java
private static String normalizeMemory(String s) {
    if (s == null) return "512M";
    s = s.trim().toUpperCase();
    if (s.matches("\\d+")) return s + "M";
    if (s.matches("\\d+[MG]")) return s;
    // fallback
    return "512M";
}
```

Use this when building commands: "-Xms" + normalizeMemory(instance.getXms())
