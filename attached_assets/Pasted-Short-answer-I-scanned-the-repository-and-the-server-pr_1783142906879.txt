   - Why it breaks: if you attempt a gplay/pubRelease build without those env vars, Gradle could fail or create an invalid signing config. (Debug builds should still work, but release workflows relying on env vars will fail.)

B — Concrete defects / code issues I found in the server-prototype (file pointers + explanation)
1. Missing Android equivalents vs. server-prototype (mismatch)
   - Files in server-prototype implement ServerInstance, ServerJarDownloader, ServerProcessManager etc. (server-prototype/src/...).
   - The Android app does not contain those implementations as mobile classes; the README references them but they are only present in server-prototype. This mismatch will confuse developers and cause integration problems.

2. ServerProcessManager: possible memory argument formatting bug
   - File: server-prototype/src/com/mojolauncher/server/lifecycle/ServerProcessManager.java
   - Code: buildCommand() adds "-Xms" + instance.getXms() and "-Xmx" + instance.getXmx()
   - Problem: If instance.getXms()/getXmx() are numeric values without units (e.g., "512" rather than "512M"), the java VM flags are invalid. You must ensure the memory values include units (M/G) before constructing the command.
   - Fix: validate and format values (append "M" if digits-only).

3. ServerProcessManager: executor never shutdown
   - File: same as above
   - Problem: ServerProcessManager creates an ExecutorService but never shuts it down; if many managers are created, threads may leak.
   - Fix: provide a shutdown hook (shutdownNow) when instance is removed/stopped permanently.

4. ServerProcessManager: logWriter concurrency and null-safety
   - File: same
   - Problems:
     - logWriter is used from streamOutput thread; writeToLog doesn't synchronize on closing/setting logWriter, possible NPE or partial writes on race with closeLogWriter().
     - openLogWriter swallows IOException; if it fails, you still continue without handling that state better.
   - Fix: synchronize around logWriter or use a thread-safe logging queue, and reliably set status if log opening fails.

5. Stream read exception handling uses process.isAlive() without null-check
   - File: streamOutput catch block — if process is null (unlikely), calling process.isAlive() could NPE. Not typically triggered, but defensively check process != null.

6. buildTail uses ArrayDeque with initial capacity argument but ArrayDeque does not strictly enforce size — not a bug but be explicit with logic. (Minor)

7. Status transitions: start() sets Status.STARTING before openLogWriter/start process; if openLogWriter or pb.start() throws, instance may remain STARTING. (state rollback needed on exception)
